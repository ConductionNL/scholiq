# opsx-reverse-spec — Learnings

Dated, atomic entries. One insight per bullet. Consolidate to SKILL.md rules at ~80 entries.

## Patterns That Work

- 2026-04-28 (openregister, large-scale Bucket 2b run): Cross-cap task refs are the primary annotation tool for misclassified Bucket 2b clusters. On openregister ~70 files across `lib/Service/`, `lib/Controller/`, `lib/BackgroundJob/` etc. all mapped to existing specs via cross-cap refs. Only 2 out of 15+ Bucket 2b clusters needed new capability specs (`object-lifecycle`, `calendar-integration`). Default should be: can I anchor this to an existing task? Only create a new spec if the answer is no.
- 2026-04-28 (openregister): One cross-ref ghost change (`bucket-2b-cross-ref`) with one task per referenced spec is a clean anchor for large-scale Bucket 2b annotation sessions. Tasks hold notes like "cross-ref: these files map to {capability}#{task}" rather than new REQs. Archive the ghost change at the end of the session like a normal retrofit change.
- 2026-04-28 (openregister): When a large Bucket 2a cluster (> 50 methods) turns out to be a scanner misclassification, the right recovery is to create a genuinely new `--cluster` spec that matches what the code actually does, then use cross-cap refs for the subset that belongs to OTHER existing specs. On openregister: "object-interactions" (252 methods) became `object-lifecycle` (5 REQs covering the actual CRUD pipeline) plus cross-cap refs to `rbac-scopes`, `linked-entity-types`, `audit-trail-immutable`, etc.
- 2026-04-21: Private helpers inherit the REQ tag of their public caller — annotate them, but don't give them their own REQ. Cuts REQ inflation (contact-matching cluster had 5 public + 5 private = 5 REQs, not 10).
- 2026-04-21: When the cluster's public methods correspond 1-to-1 with distinct observable behaviors, the 5-REQ cap is rarely binding. Cluster sprawl only happens when a single public method wraps multiple behaviors internally (orchestrators are the common case).
- 2026-04-24: The skill generalises cleanly to frontend/JS clusters. The `@spec openspec/changes/.../tasks.md#task-N` convention works identically in JSDoc blocks as in PHPDoc — place tags at the same relative position (after `@param` / `@return`, before the closing `*/`). File-header tags go in the top-of-file docblock. Confirmed on mydash's `legacy-widget-bridge` cluster (5 JS methods in `src/services/widgetBridge.js`).
- 2026-04-24: 5 methods ≠ 5 REQs. Two read-only query methods that inspect the same internal map can legitimately share a single REQ (scenario-per-method inside one REQ). On `legacy-widget-bridge`, `hasWidgetCallback` + `getRegisteredWidgetIds` collapsed into REQ-LWB-004 because they're the same "inspect registration state" capability. Cluster with 5 methods / 4 distinct behaviours → 4 REQs is the right answer; the cap-of-5 is an upper bound, not a target.
- 2026-04-24: Construction-time side effects with observable external contracts (global monkey-patches, event-bus subscriptions, DI-container registrations) deserve their own REQ — do NOT fold them into "constructor is plumbing". Test: can a caller outside the class observe the side effect? If yes → REQ material. On `legacy-widget-bridge`, `interceptRegistration()` is called only from the constructor but monkey-patches `window.OCA.Dashboard.register` — the external contract IS the capability's core behaviour.
- 2026-04-24 (app-versions, small-app run): Format-matching works across spec conventions. app-versions uses `### Requirement: Title [MVP]` (forward-building feature-spec format) rather than `### REQ-NNN` (retrofit format). The skill's "match sibling voice/format/granularity" guardrail held up — the retrofit delta used the Requirement-with-tier form so the capability stayed internally consistent after archive-merge. `retrofit_extensions` frontmatter was keyed by the Requirement title string (`retrofit_extensions: ["Frontend Context Endpoints"]`) rather than a numeric REQ id. → ✅ Promoted to SKILL.md Step 8 (2026-04-24).
- 2026-04-24 (app-versions): `--extend` bias pays off on tiny apps. Two unspecced endpoints (`adminCheck`, `updateChannel`) folded cleanly into ONE new Requirement under the existing `version-management` capability. Creating a separate `frontend-context-endpoints` capability would have fragmented a 5-file codebase. The default bias is the right default even when `--cluster` looks plausible.
- 2026-04-24 (nextcloud-app-template): A retrofit can legitimately **fill in an existing referenced change** rather than minting `retrofit-X-DATE`. The template had `@spec openspec/changes/example-change/tasks.md#task-N` tags in 10 files but no `example-change/` directory — so 15 `@spec` tags were dangling. The right move was to populate the referenced change (proposal + tasks + design + 7 capability specs marked `example: true`) rather than add a parallel retrofit change. The skill doesn't currently model this case; it falls between "retrofit ghost change" and "forward-building change". → ✅ Promoted to SKILL.md Step 1 (2026-04-24) — detection only; fill-in vs. parallel-retrofit decision held in learnings (n=1, template-specific).

## `--cluster` decision

- 2026-04-24: When choosing `--cluster` over `--extend`, add a "Why a new capability rather than extending {nearest-existing-capability}" section in `proposal.md`. Two-to-three sentences naming the nearest candidate and why this cluster is not an extension of it. Reviewers will ask; pre-empting the question makes the call auditable and keeps the default bias-toward-`--extend` honest. Example: on mydash legacy-widget-bridge, the nearest candidate was `widgets`, rejected because the bridge is purely about the legacy `window.OCA.Dashboard.register` callback pattern while `widgets` covers modern `IAPIWidget`/`IAPIWidgetV2` discovery, placement, and item loading.

- 2026-05-01 (openregister, actions --cluster): REQs for a standalone entity system (Actions) can be grouped by lifecycle phase rather than by file/class: (1) CRUD management, (2) trigger mechanism, (3) execution mode, (4) failure handling, (5) scheduling. This 5-REQ shape maps cleanly to the Actions feature surface and keeps the spec readable without splitting into micro-REQs per method.
- 2026-05-01 (openregister, oas-generation): A 2-method cluster where both methods call the same underlying service (with/without a scope parameter) still warrants 2 separate REQs when the observable contracts differ: different inputs, different output shape (register-specific `info` section), different API endpoints. Don't collapse into 1 REQ just because the implementation is nearly identical.
- 2026-05-01 (openregister, actions): When a method (`calculateDelay`) is clearly implemented but never called in the expected code path, flag it in the REQ notes section rather than writing the REQ as-if the behavior is fully working. The spec then becomes an honest record — it describes what was intended AND surfaces the gap. This is more useful than either silently ignoring the unused method or writing an aspirational REQ.
- 2026-05-04 (openregister retrofit audit): Frontmatter format inconsistency hides cohort flags from Specter. Inline `[REQ-005]` and block `- REQ-005` parse identically, but ad-hoc inline drift on hand-edited specs (or full-text values like `["The system MUST..."]` left over from the pre-2026-04-24 convention) made post-hoc auditing miserable. SKILL.md updated to mandate block YAML for `retrofit_extensions` and bare REQ-IDs (no quotes, no full text). Block also makes per-item comments possible if reviewers want to annotate individual REQs.
- 2026-05-04 (openregister retrofit audit): Older retrofit ghost changes (2026-04-28 batch) lack `design.md` because the skill's step 11 (`/opsx-ff` to fill in design.md) post-dates them. The SKILL.md is current, but a one-time backfill is needed for any pre-step-11 retrofit changes — write `design.md` directly into the archived ghost change folder (the playbook's "textual reference" semantics mean the path doesn't need to be physically resolvable to be useful for human readers).

## Mistakes to Avoid

- 2026-04-21: Do NOT write REQs against `private function` signatures — inputs/outputs are implementation detail. Look at the public caller's contract and let the helper inherit its REQ tag.
- 2026-04-21: Do NOT assume the coverage report's branch is `development`. Apps like decidesk keep specs on `beta`; always read `coverage-report.json.branch` and check out from there.
- 2026-04-24: Do NOT trust the coverage-report blindly. If the classifier miscategorised Bucket 1 as Bucket 2 (e.g. because the class name didn't contain the owning capability's keyword — seen on mydash: `RuleEvaluatorService` / `VisibilityChecker` / `UserAttributeResolver` were initially Bucket 2b but REQ-VIS-005..010 in `conditional-visibility/spec.md` covered them by scenario content), reverse-spec will mint duplicate REQs. Before step 6 (Draft the REQ(s)), grep sibling specs for the cluster's method and file names. If any existing scenario references them, stop and re-verify with the user — the cluster may be Bucket 1 in disguise.
- 2026-04-24: Do NOT forget `.git-blame-ignore-revs` at the end of the run (step 15 sub-bullet). Easy to miss because step 15 reads as "commit structure". Make it an explicit sub-step for the second commit: `echo "$(git rev-parse HEAD)" >> .git-blame-ignore-revs` right after the archive commit lands.
- 2026-04-24 (nextcloud-app-template): A task's `@spec` tag can legitimately span files implementing **unrelated capabilities** (template task-5 annotates `ItemController`/`ItemService` which demonstrate ADR-005 per-object auth AND `Repair/InitializeSettings` which is a first-install repair step). Do NOT silently treat these as one cluster — the two bodies of code are in different capabilities and need different REQ homes. Surface the cross-capability span during step 3 (load the cluster) and decide: split into two tasks, or document the overlap in design.md and implement REQs across both capabilities under the single task. The template case chose the second path because editing the docblocks was out of scope; future runs against non-template apps should prefer splitting.
- 2026-04-28 (openregister "object-interactions"): Do NOT proceed with a Bucket 2a cluster that has > 50 methods without first verifying alignment between the cluster's class names and the owning capability's REQ scenarios. On openregister, 252 methods were labeled "object-interactions" (adjacent to `notes-tasks-files` spec) but were actually core CRUD lifecycle handlers (`SaveObjects`, `ValidateObject`, `CacheHandler`, etc.). Proceeding would have minted 5+ duplicate REQs. Rule: when cluster method count > 50 OR class names don't match the capability keyword, grep the owning spec's scenario bodies for class/method names before drafting any REQ.
- 2026-04-28 (openregister): Do NOT use background agents for annotation batches of 70+ files. Background completion cannot be monitored and partial annotation leaves the branch in ambiguous state (some files annotated, others not, no way to distinguish which ones). Use foreground agents for large batches; split into ≤ 40-file sub-batches if needed.

## Domain Knowledge

- 2026-04-21: OpenRegister's `ContactMatchingService` is a canonical `--cluster` candidate (Bucket 2b): cohesive, no existing capability spec, 5 public methods across 5 distinct behaviors (email/name/org matching, orchestrated matcher, cache invalidation). Use it as the reference shape for future `--cluster` runs.
- 2026-04-21: Magic numbers that are observable via return values (e.g. ContactMatchingService's 1.0 / 0.8 / 0.7 / 0.4 confidence scores) DO belong in REQ language. If the implementation re-tunes them later, the spec delta follows. This is not "aspirational" — it's "observable".

## Open Questions

- 2026-04-21: When a Bucket 2a cluster's REQ numbering needs to slot into an existing capability that already uses non-sequential numbering (e.g. gaps from archived deletions), should `/opsx-reverse-spec` fill the gaps or always append? Current behavior is append — confirm this matches reviewer expectations after 2-3 more `--extend` runs.
- 2026-04-24 (app-versions): When the existing spec has stale frontmatter (`status: idea` despite the code being implemented), should reverse-spec surface this as a side finding? It's adjacent to the skill's scope — not a REQ-drafting problem per se, but a signal the capability is further along than its metadata suggests. Current behavior is "silently ignore and get on with drafting". Candidate for step 5 (load architectural context): if the target capability is `status: idea` but the code exists, flag and ask whether to also bump status.
- 2026-04-28 (openregister Bucket 2b): Bucket 2b cluster names on large apps reflect `lib/{category}/` directories, not observable behaviors. Most methods in `lib/Service/`, `lib/Controller/`, `lib/BackgroundJob/` etc. map to existing specs via cross-cap refs. The skill's cluster-selection step should surface this expectation upfront: when a Bucket 2b cluster name is a filesystem namespace (`service`, `controller`, `event`), expect extensive pre-analysis before knowing which existing spec each method maps to — this is cross-cap annotation work, not `--cluster` spec-creation work.
- 2026-04-28 (openregister): When a single Bucket 2b annotation session covers 70+ files mapping to 20+ different existing specs, one cross-ref ghost change (e.g. `bucket-2b-cross-ref`) with one task per spec is a valid anchor pattern. This does NOT mint new REQs — it maps each task to an existing spec's task. The skill's step 7 should acknowledge this as a valid ghost change shape alongside the normal retrofit scaffold.

## Consolidated Principles

- (promoted after 3+ confirmations across runs)
- **Match sibling spec voice/format/granularity** — confirmed across 3 runs on 3 different apps:
  - mydash uses `### REQ-LWB-NNN` style → delta used same
  - app-versions uses `### Requirement: Title [Tier]` style → delta used same
  - nextcloud-app-template (multi-capability fill-in) matched the `### Requirement: ...` convention that its own `openspec/specs/README.md` documents
  Every mismatch would leak a house style into the app. Candidate for promotion into SKILL.md step 5 as a named sub-step ("Voice match: find at least one sibling capability spec; your delta's heading style, scenario syntax, and tier/status markers MUST mirror it").
