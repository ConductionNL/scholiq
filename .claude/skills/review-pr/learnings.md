# Learnings — review

---

## Consolidated Principles

These have been confirmed across multiple PRs or are explicit workflow rules. Apply every time.

- **Verify group completeness**: when patching a set of similar methods or fixing a bug class, check every other occurrence in the diff. The same PR that correctly fixes `executeQuery()` in one mapper can silently repeat the bug in a migration file. Ask: "where else in this PR's diff does the same pattern appear?"
- **One comment per finding**: never combine concerns. Authors can only resolve individual GitHub review comments, not paragraphs.
- **Diff lines only**: every comment target line must fall within a diff hunk's `[new, new+len)` range. Lines outside the diff cause 422 errors. Compute new-file line numbers step by step: context lines and added lines both increment the counter; removed lines (`-`) do not.
- **Always verify deferral and author claims**: when an author defers a concern as "pre-existing" or "out of scope," grep the diff to confirm the change site is not in the PR. When a subagent escalates a finding based on inferred semantics (e.g. GitHub Actions `if:` logic), trace the expression manually before accepting it as a blocker. Confirmed across 3+ PRs: CHANGELOG vs diff mismatch, "pre-existing" deferral inaccuracy in openconnector#733, subagent `if:` inversion in openconnector PR.
- **Step 4 (strictness mode) is mandatory**: always use AskUserQuestion to confirm strictness before running analysis — even for re-reviews and tiny fix commits. The recommendation (Quick/Standard/etc.) comes from the signals, but the question must be asked. Skipping it removes the reviewer's control over review depth.
- **Verdict body must link to inline comments**: after posting the COMMENT review (Step 8), fetch `html_url` for each new inline comment (`gh api .../pulls/{n}/comments --jq '[.[] | {id, html_url}]'`) and include markdown links in the Step 9 REQUEST_CHANGES/APPROVE body. Authors read the verdict first — plain-text references force them to hunt for the comment.
- **`gh pr checks --json` is unreliable — always use the check-runs API**: `gh pr checks --json name,state,conclusion` silently returns `[]` when GitHub hasn't indexed the checks under that schema. Always fetch via `gh api repos/{owner}/{repo}/commits/{sha}/check-runs`. Never treat an empty JSON array as "all checks pass."
- **Re-review: commit message is the primary resolution signal**: commit messages in a fix commit often enumerate which review comments were resolved by name or number. Parse them first before re-reading the full diff. Thread replies are supplementary.
- **GitHub review body update uses PUT, not PATCH**: `PUT /repos/{owner}/{repo}/pulls/{pull_number}/reviews/{review_id}` with `{"body": "…"}` — PATCH returns 404. Only works for submitted (non-pending) reviews with a `body` field change; the state cannot be changed via this endpoint.
- **Check PR description before reviewing (Step 2a)**: always read `body` from `gh pr view --json body` in Step 2. If it is empty or fewer than 50 non-whitespace characters AND the PR is non-trivial (`changedFiles > 1` or `additions > 30`), post a top-level issue comment asking the author to add a description, then ask the user whether to stop or continue. Reviewers cannot verify intent against the diff without it; missing context is the root cause of many false approvals and false blockers alike. **Exception — branch-promotion PRs** (e.g. `development → beta`, `beta → main`, `main → production`): a missing description is normal and expected. Instead of flagging it, derive intent from three sources: (1) check whether `CHANGELOG.md` was modified in the PR and read those entries; (2) read the commit messages (and their bodies) of every commit in the PR — they carry the feature/fix context; (3) scan commit messages for `Merge pull request #N` patterns and fetch those PR titles/bodies with `gh pr view N --repo {owner}/{repo}` to get the original PR descriptions. Use this aggregated context as the analysis brief instead of the PR body.
- **Branch protection: always check rulesets, not just the classic API**: `GET /repos/{owner}/{repo}/branches/{branch}/protection` returning 404 does NOT mean the branch is unprotected. ConductionNL (and many other orgs) use GitHub's newer ruleset system — always also query `GET /repos/{owner}/{repo}/rulesets` before concluding protection is absent. A false 404 on the classic endpoint is the most common cause of a spurious security REQUEST_CHANGES on CI/CD workflow PRs.
- **CI/CD workflow trigger semantics require explicit before/after comparison**: when a PR replaces `pull_request: types: [closed]` + `if: github.event.pull_request.merged == true` with `push`, the gate changes from "a PR must exist and be merged" to "any push, including direct commits." Always compare old and new trigger semantics side by side, then evaluate branch protection (both classic API + rulesets) to determine whether the review gate survives the change. The three signals compound each other: `push` trigger + `secrets: inherit` + no branch protection = open secret exposure on any write-access push.

## Patterns That Work

- 2026-04-24: Spawning a subagent for analysis and passing the diff file path (not the raw content) keeps the main context clean and allows the subagent to explore specific hunks on demand. For diffs >50KB this is essential.
- 2026-04-24: Posting all inline comments in a single `POST /reviews` payload (event: COMMENT) with a `comments[]` array is more reliable than posting comments one by one. It avoids race conditions and lets GitHub validate all line positions at once.
- 2026-04-24: Separating the inline-comment review (event: COMMENT) from the verdict review (event: REQUEST_CHANGES or APPROVE) produces a cleaner PR timeline. Mixing them in one call causes the verdict to appear before the comments in the UI.
- 2026-04-24: For CI/CD workflow changes that remove `synchronize` from `pull_request.types`, verify the `push` branch filter covers all realistic source branches. If it only covers named patterns (`feature/**`, `bugfix/**`), dependabot/renovate branches fall through — always flag this as a concern.
- 2026-04-28: When a PR enables a previously-dormant code path (e.g. fixing a JOIN that returned zero rows), pre-existing logic bugs in that path become live. Always check loop control flow (`break` vs `continue`) and null-guards in any code the PR's fix causes to execute for the first time.
- 2026-05-05: For docs-only OpenSpec PRs (only `openspec/changes/` and `openspec/specs/` modified), the right review framing is workflow-conformance + cross-doc consistency, not code correctness. The three blocker classes that recur: (1) canonical spec at `openspec/specs/<name>/spec.md` committed in the same PR as the change-delta — canonical specs are produced by `/opsx-archive` after implementation, not in the proposal PR; (2) `plan.json` shipped at proposal time — that artefact is generated post-approval by `/opsx-plan-to-issues`; (3) PR title vs content drift, especially when the title names a parent project ("anonymisation project") that the proposals/specs never reference. Make these three the first checks on any spec-only PR.
- 2026-05-05: PR-level findings (title↔content drift, PR-description issues) don't have a sensible inline anchor. Post them as a top-level issue comment via `gh api repos/{owner}/{repo}/issues/{N}/comments` rather than forcing them onto a random file:line. Then link the issue-comment URL from the verdict body alongside the inline-comment URLs.
- 2026-05-05: For re-reviews where the author pushed a fix commit AND merged the base branch in, `compare(prior_head, new_head)` returns hundreds of unrelated files from the base merge. Use `compare(fix_commit_parent, fix_commit)` to isolate the author's intentional changes — that's the actual surface to verify against the prior findings. Verified on openregister#1427: prior_head→new_head returned ~330 files of dev-branch noise; fix_commit_parent→fix_commit returned the 11 files the author actually changed.
- 2026-05-05: GitHub's `commits/{sha}/check-runs` API returns BOTH old runs from the base commit AND new runs from the PR head, deduplicated only by name. To get the *current* status, fetch with the `check_name` filter or post-process with `group_by(.name) | map(max_by(.conclusion // ""))` — otherwise stale "failure" entries from earlier commits will appear alongside the latest "success" runs and trigger a false CI banner in the review.
- 2026-05-05: For massive PRs (>300 files), `gh pr diff` returns HTTP 406 ("PullRequest.diff too_large"). Fall back to a shallow clone — `git clone --depth=50 --branch <base> <repo>` then `git fetch --depth=50 origin pull/N/head:pr-N`. That gives full diff access via `git diff origin/{base} -- <path>` and lets `scripts/run-gates-on-pr.sh` run from the clone. ~30s on openregister-scale repos.
- 2026-05-05: Files in the GitHub `/pulls/{n}/files` API list ≠ files visible in `git diff`. A file with net deletions only (no `+` lines) is filtered out of the API but DOES show in `git diff` and may appear in mechanical-gate output. The 422 error message at posting time is "Path could not be resolved". Resolution: anchor the inline comment on a closely-related file that IS in the API list (e.g. `lib/Service/ActionService.php` instead of `lib/Controller/ActionsController.php` when only the controller is deletion-only) and explain the redirect in the comment body.
- 2026-05-05: For large security-sensitive PRs, dispatching a code-review subagent and a persistence-audit subagent **in parallel** (single Agent tool call with two invokes, both `run_in_background: true`) cuts elapsed time roughly in half vs serial. Brief each agent explicitly NOT to duplicate the other's scope — persistence agent on token / audit-trail / webhook / CI-secrets / lifecycle vectors; code agent on standard correctness + RBAC + injection. Overlapping findings on the same file:line are resolvable by consolidating into a single inline comment with both framings.
- 2026-05-05: When mechanical-gate findings overlap with the analysis agent's scope, instruct the agent explicitly NOT to re-discover them — feed the gate findings into the brief and ask the agent to focus elsewhere. Without this, you get duplicate findings on the same lines, and the gate findings drown in the inline-comment volume. Post mechanical-gate findings as a single dedicated top-level summary instead of one inline per gate hit (especially for high-volume gates like no-admin-IDOR with 185 hits where most are pre-existing-in-modified-files).
- 2026-05-05: Strict-mode reviews on PRs >300 files can produce 30+ findings. Cap the agent at ~20-25 inline comments and overflow off-diff findings to top-level issue comments — authors stop reading after ~30 inline notifications. The `REQUEST_CHANGES` verdict body still references every finding via markdown links, just not every line as inline.

## Mistakes to Avoid

- 2026-04-30: When a REQUEST_CHANGES review is dismissed, GitHub appears to auto-resolve the inline comment threads that belonged to it. Always fetch thread `isResolved` state after a dismissal before attempting to resolve threads manually — they may already be done.
- 2026-04-24: `line: null` in `GET /pulls/{n}/comments` does not mean a comment failed to attach. GitHub converts `line` to `position` internally. Verify by checking the comment ID exists, not `.line`.
- 2026-04-24: `gh api .../reviews --input file.json` returns `comment_count: 0` even when all inline comments land correctly — verify success by checking `.../pulls/{n}/comments` separately.
- 2026-04-24: Deleted integration tests replaced by a pointer comment are a regression risk. Always read the full context of removed test code — if a removed test covered a specific edge case, check whether that case is tested elsewhere.
- 2026-04-28: PHPUnit mock `method()->with($x)->willReturn($y)` silently returns `null` when called with a non-matching argument (no `expects()` set). A test configured `->with(self::UUID_CONSTANT)` but production code passing `getId()` (integer) returns `null` and causes a `TypeError` at `foreach(null)` — always verify mock expectations match the actual parameter type.

## Domain Knowledge

- 2026-04-24: ConductionNL/openregister has parallel SQL and PHP paths for RBAC: the SQL emitter (`applyRbacFilters`, `buildRbacConditionsSql`) handles the list endpoint; a PHP-side matcher (`ConditionMatcher`, `PermissionHandler`) handles find/single-object. Drift between these paths is the root cause of most list-vs-find inconsistencies.
- 2026-04-24: PHP's loose comparison coerces `null` to `""` (string) or `0` (numeric) for `<`, `>`, `<=`, `>=`. `null <= "2026-01-01"` returns `true` in PHP; SQL `NULL <= '2026-01-01'` returns NULL (filtered). Any comparison operator without an explicit null guard is a parity bug.
- 2026-04-24: PHP's array union operator (`+`) keeps the left-hand key on collision; `array_merge` overwrites with the right-hand value. Non-obvious precedence deserves a comment.
- 2026-04-24: SQL `col = NULL` (from `quoteValue('NULL')` in query builders) is always false/unknown — the correct SQL form is `IS NULL`. Pre-existing pattern in ConductionNL/openregister's SQL emitter.
- 2026-04-24: Symfony embeds Nextcloud route `requirements` verbatim inside named PCRE capture groups. A `^` inside a requirement anchors to mid-string and never matches — omit it. The pattern `(?!api(?:/|$)).+` is the correct form to exclude an API prefix.
- 2026-04-24: In ConductionNL/openconnector's `appinfo/routes.php`, all API routes use a lowercase `/api/...` prefix. A case-sensitive negative lookahead `(?!api(?:/|$))` covers all of them.

## Open Questions

- 2026-04-24: For PRs that touch both a new adapter class AND a service it delegates to, check both the adapter's null guards AND whether the adapter's interface contract (e.g., `$validate`) is honored or silently discarded.
- 2026-04-24: When a `@deprecated` tag says "no production callers," verify the claim extends to all consumer repos in the org — consumer apps that call entity methods directly are invisible to a grep of the source repo.
- 2026-04-24: `new DateTime()` timezone trap: always flag PHP `$now` computations without a timezone. Dutch government deployments commonly run CET/CEST servers against UTC databases.
- 2026-04-30: For extraction PRs (shared service extracted from duplicate private methods), always check: (a) whether the new service handles all edge-case inputs both prior implementations handled differently; (b) whether handler-level tests still enforce the delegation path, not just constructor wire-up. A service with 55 unit tests but no tests through the caller means a delegation resignation goes undetected.
- 2026-04-30: `$cache[$key] ?? fallback` pattern risk: when a batch cache uses `null` as "not initialized" and `[]` as "no items found", a cache-miss on a sub-entity UUID that was never batched collides with "entity has zero items" — use `array_key_exists` to distinguish the two states.
- 2026-05-01: BREAKING CHANGEs that knowingly defer test updates ("integration tests deferred to a follow-up") create a pattern where development→beta merges always arrive with failing CI. Consider whether the team wants a policy requiring test fixtures to be updated in the same PR as the BREAKING CHANGE, or whether an explicit exception process is needed.
