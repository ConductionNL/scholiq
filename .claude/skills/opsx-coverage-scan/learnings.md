# opsx-coverage-scan — Learnings

## Patterns That Work

- 2026-04-20 (procest, full pass): CamelCase-aware tokenization with noise-word stripping (Controller/Service/Helper/...) is essential — without it, method `index()` in `MetricsController.php` fails to match REQ-PROM-001 "Metrics Endpoint" because `metricscontroller` is one token. Splitting on lowerUpper + ACRUpper boundaries fixed scoring across ~200 controller methods.
- 2026-04-20 (procest): Encoding the pilot's judgment calls as an explicit `(file, method) -> (capability, req_id, confidence, signal)` override table is the right shape for the classifier. The table captures things that keyword scoring cannot recover — e.g. `ZrcController = ZRC API = REQ-003`. Keyword scoring is the fallback for everything else.
- 2026-04-20 (procest): Conservative Bucket 3b — only REQs in capabilities with **zero** Bucket 1 methods count as unimplemented. Listing every REQ that didn't get a confident method match would have produced 290 false positives; the conservative rule yielded 73 honest unimplemented REQs across 19 untouched capabilities.
- 2026-04-24 (app-versions, tiny-app run): On apps with ≤ 10 code units + ≤ 1 capability spec, the "single main-context pass" model works fine. Subagent delegation with scratch files (the openregister pattern) is overkill at this scale — the added coordination friction costs more than it saves. Observed anchor: ≤ 10 units inline works; 703 units (openregister) needs subagent. Candidate threshold (needs a data point in the 50–200 range before promoting to SKILL.md): delegate when `code_units > 100 AND REQs > 50`, otherwise classify inline.
- 2026-04-24 (app-versions): Scenario-body grep beats class-name-keyword match even when class names DO share the capability keyword. app-versions' `ApiController::apps` / `appVersions` / `installVersion` all matched existing Requirements not by class-name similarity but because each Requirement's scenario text ("admin views installed apps", "view versions for an app", "install an older version") contains the same verb+noun tokens as the method names. Supports the enrichment proposed in #177: scenario-body grep as the PRIMARY signal, not a fallback.

- 2026-04-28 (openregister, real run + explore session): The single-pass reverse-scan approach is ~40× faster than per-REQ git log -S on a 7k-commit repo: `git log --all -p -- lib/ src/ | grep "^-[^-]"` takes 7s and produces ~946k removed lines (42MB); subsequent `grep -i` per REQ runs in ~20-50ms. Per-REQ git log -S costs ~2.8s/call × 384 REQs = 18+ minutes. Single-pass also captures removed LINES directly, which is semantically more precise for "was this code ever removed?" than the pickaxe (which finds commits where string count changed, catching both additions and removals).
- 2026-04-30 (openregister): `status: redirect` (and similar frontmatter values) in a spec.md means the spec legitimately has 0 REQs — it points to an owning app. Surfacing these as "parser missed REQs" is a false signal. The correct treatment is: detect the status field in Step 3, skip REQ extraction, and emit a Bucket 3b entry with a `status:redirect` note. Confirmed in 2 runs (2026-04-23 + 2026-04-30). (promoted from learning-candidates.md 2026-04-30)
- 2026-04-28 (openregister): Current-codebase grep is the WRONG fallback for the reverse pass. "keyword in current code but not matched in Pass A" ≠ "implementation existed and was removed." If the single-pass cache build times out, collapse ALL unimplemented REQs to 3b with a note — do not grep current code.

## Mistakes to Avoid

- 2026-04-28 (openregister, Bucket 2a "object-interactions"): A cluster named after a capability keyword (`object-interactions` → adjacent to `notes-tasks-files`) can be a massive misclassification. On openregister, 252 methods from `lib/Service/Object/` were labeled "object-interactions" by the scanner because the directory name contains "Object" — but the actual capability the code implements is a CRUD lifecycle pipeline, not the notes/tasks/files spec at all. **Rule**: when a Bucket 2a cluster has > 50 methods, read 5-10 random method names and ask "do these match the owning capability's REQ scenario language?" before emitting the cluster. If the answer is clearly no, re-cluster by class-name stems and emit to a new capability, or flag for human pre-split.
- 2026-04-28 (openregister, Bucket 2b): Bucket 2b cluster names reflect `lib/{subdir}/` directory structure, not observable behavior. Clusters named `service`, `controller`, `background-jobs`, `event`, `tool`, `appinfo` are useless for reverse-spec without a human pre-analysis step. **Improvement candidate**: when a Bucket 2b cluster name is a filesystem namespace word (matches `service|controller|background|job|event|tool|appinfo|middleware|exception`), emit a warning in the report: "Cluster name is a namespace word, not a behavioral name. Pre-split required before running `/opsx-reverse-spec`." This saves the false-start of opening a `--cluster service` reverse-spec run.
- 2026-04-20 (procest): Initial keyword scoring at threshold 0.70 produced **zero** Bucket 1 matches — the scorer was too strict because: (1) `noise` words like "Controller" weren't stripped, (2) no CamelCase split, (3) no bonus for method-name vs. path-only overlap. The 0.70 threshold itself is fine; the signal calculation underneath needs all three features.
- 2026-04-20 (procest): The spec heading parser must accept `####` (not just `###`) — `zgw-business-rules-compliance/spec.md` uses `#### ZRC-007: ...` and was silently dropped by a `^###\s` regex that looked correct.
- 2026-04-20 (procest): Compound REQ IDs like `ZRC-005b/023h` or `ZRC-016/018/019/020` are real in procest's delta specs. A strict `REQ-[A-Z]+-\d+` regex drops them. Accept `[A-Z]{2,4}-[0-9]+[a-z]*` and don't try to split the compounds — keep them as a single REQ.
- 2026-04-24 (mydash): Class-name ↔ capability-name matching produces Bucket 2 false positives when services are named by behaviour-noun (`RuleEvaluatorService`, `VisibilityChecker`, `UserAttributeResolver`) while the owning capability is named by domain-noun (`conditional-visibility`). On mydash, three services were initially flagged Bucket 2b even though REQ-VIS-005..010 in `conditional-visibility/spec.md` cover them verbatim by scenario content. Root: Pass A scored class-name vs. capability-dir-name but never scored class-name or public-method-name against REQ scenario bodies. **Fix: before declaring Bucket 2, grep every `openspec/specs/**/spec.md` scenario body for the class name AND each public method name. A direct mention re-pins to Bucket 1 (with NEEDS-REVIEW since the signal is indirect). Cost note: on large apps (procest: ~200 methods × 43 specs) this grep adds real work to Pass A — when promoting to SKILL.md, gate it: only run scenario-body grep when the class-name-vs-capability-dir score falls below 0.70, rather than unconditionally for every class.**
- 2026-04-24 (nextcloud-app-template): The skill's step 1 guardrail ("If specs are empty, stop and suggest `/app-explore` + `/app-design` first") fires correctly when `openspec/specs/` has only a README, but the advice is incomplete when the app ALSO has dangling `@spec openspec/changes/{change-name}/tasks.md#task-N` tags in its `lib/**/*.php`. Those tags signal partial scaffolding with a human intent attached (someone wrote the tags expecting a change to exist later). "Stop and run `/app-explore`" erases that intent. Better: detect the dangling-tag case and recommend filling in the referenced change (via reverse-spec with a new flag, or via a separate scaffold skill). Flag, don't halt.
- 2026-04-24 (app-versions): When an app's existing spec uses `status: idea` frontmatter but the code is fully implemented (app-versions' `version-management/spec.md` is `status: idea` despite 4 Requirements being live in code), the scan report should surface this as a Notes-section finding. Unclear if it deserves its own bucket or stays a free-text note — current behaviour is silent. Candidate class name: `stale-spec-status`. Worth watching across 2-3 more apps before codifying.

## Domain Knowledge

- 2026-04-20 (procest): `case-management/spec.md` has duplicated REQ blocks (lines 63–945 then 1013–1946). The counter reports 45 REQs; only ~22 are unique. Pre-retrofit cleanup recommended before `/opsx-annotate` runs against this spec.
- 2026-04-20 (procest): `zgw-business-rules-compliance` is a delta spec with 11 ZRC-specific REQs, but procest ships ZgwDrcRulesService, ZgwZtcRulesService, ZgwBrcRulesService with the same pattern. These files land in Bucket 2a (extend the capability) rather than 1 — the spec simply doesn't cover DRC/ZTC/BRC rules yet.
- 2026-04-20 (procest): Procest's large ZGW controller files (ZrcController 39 methods, DrcController 32, ZtcController 21, BrcController 18) all fit the "one controller = one ZGW API = one REQ-00N in zgw-api-mapping spec" pattern. Per-method assignment under that REQ is deferred to `/opsx-annotate` because the spec's REQ-001..REQ-021 don't split cleanly per REST verb.

## Open Questions

- 2026-04-20 (procest): For frontend Vue components, should per-method classification try harder (read `methods:` block bodies, match against REQ scenarios) or is file-level "inherit capability's first REQ with NEEDS-REVIEW" the right fidelity? For procest, 286 FE units all landed at 0.78 confidence — useful for annotating at file level but shallow at method level.
- 2026-04-20 (procest): `workflow-import-export` spec exists (2 REQs) and procest ships `CaseDefinitionController + CaseDefinition{Export,Import}Service`. Classifier kept them in 2a — is it Bucket 1 (export/import of workflow templates) or 2a (case definitions are different from workflow templates)? Need a human read of the spec body vs. the controller to decide.
- 2026-04-20 (procest): Bucket 1 at 678 methods exceeds the "chunk by capability" guidance (150-method threshold). If `/opsx-annotate` doesn't grow a `--capability` flag, this will produce one very large ghost-change PR that's hard to review. Recommend the scanner emit a capability-list hint the annotator can consume.
- 2026-04-23 (openregister): On apps this size (703 code units, 384 REQs, 43 specs), delegating the entire 10-step scan to a single general-purpose subagent with intermediate `/tmp/` scratch files produces a clean output in one pass. Main-context classification would have been context-starved. Pattern: main agent sizes the scan → asks user once → dispatches one subagent → verifies outputs + git-clean + valid JSON on return. Total duration ~107 minutes; sub-400-word return kept the consolidation cheap.
- 2026-04-23 (openregister): Bucket 2b clustered by raw directory name (`service`, `modals`, `controller`, `views`, `store`) produced mega-clusters of 490-1133 methods. These aren't reverse-spec-able as a unit — the cluster label says nothing about observable behavior. Improvement candidate: when a 2b cluster exceeds ~30 methods, re-cluster its members by shared method-name verbs or class-name stems before emitting. Otherwise every 2b entry requires a human pre-split before `/opsx-reverse-spec --cluster` is usable.
- 2026-04-23 (openregister): 782 plumbing methods (vs. 649 Bucket 1) reflects heavy magic-method + thin-delegator usage. The plumbing detector working at this scale is reassuring — but the ratio means `/opsx-annotate` will see a much smaller annotation surface than the file count suggests. Useful prior for estimating retrofit cost on similar-shaped apps.
- 2026-04-23 (eval-surfaced, see `evals/grading.json`): Baseline 0.66 across 4 evals. Hard 0.00 on `large-app-asks-confirmation` — the Large-app warning guardrail (line 237) is stated as prose but has no corresponding step in the 10-step sequence. Same structural weakness as the sibling `opsx-annotate` idempotent-rerun eval. Also `python-exapp-deferred` scored 0.78 because the stop directive is a terse bullet inside Step 4 ("Python (ExApp wrappers only): stop — Python variant deferred.") rather than a named halting substep. → ✅ Step 4.5 large-app guard promoted to SKILL.md (2026-04-24).
- 2026-04-23 (eval-surfaced): Step 3's REQ inventory parser spec says "Extract `REQ-NNN` block" but real Hydra specs use `####` headings and compound IDs (`ZRC-005b`, `ZRC-016/018/019/020`). → ✅ Promoted to SKILL.md Step 3 (2026-04-24): heading parser now `^#{3,4}\s`, REQ ID pattern `[A-Z]{2,4}-[0-9]+[a-z]*`.

## Consolidated Principles

Validated after 3+ confirmations across runs (procest, openregister) or after resolving a measured eval failure. These are candidates for promotion to SKILL.md guardrails/rules on the next edit.

- **CamelCase-aware tokenization with noise-word stripping is non-negotiable.** Without it, `MetricsController.index()` cannot match "Metrics Endpoint" REQ. The tokenizer must split on lowerUpper + ACRUpper boundaries and strip class-suffix noise (Controller/Service/Helper/Repository/Factory/etc.). Confirmed across procest and the openregister pass — first-principles requirement for any retrofit scanner.
- ~~**REQ heading parser must accept `####` and compound REQ IDs `[A-Z]{2,4}-[0-9]+[a-z]*`.**~~ → ✅ Promoted to SKILL.md Step 3 (2026-04-24).
- **Conservative Bucket 3b (capability with zero Bucket 1 methods) over greedy (every unmatched REQ).** Procest: greedy produced 290 false positives; conservative yielded 73 honest unimplemented REQs. When the choice is between false positives in a human-triage bucket vs. true negatives, prefer the honest number.
- **Delegate large scans (> ~400 code units) to a single general-purpose subagent with scratch files.** Openregister pass (703 units) proved this: main-context classification would have been context-starved. Pattern: main agent sizes the scan → confirms with user → dispatches one subagent → verifies outputs on return. Single subagent beats fanning out because Pass B (private-helper resolution) needs cross-method context.
- **Read-only guardrail is structural, not cosmetic.** Writes only `openspec/coverage-report.md` + `coverage-report.json`. Any pressure to "just fix this while I'm here" compromises the separation between scan (decide) and annotate (execute). Downstream skills exist for exactly that reason.
