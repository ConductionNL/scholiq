## Context

Scholiq integrates with the Dutch K-12 education system (VO: voortgezet onderwijs, 3-6 years; MBO: vocational, 2-4 years). The exam administration lifecycle is annual:
- **September–October:** PTA determination (subjects, topics, weights, test moments, whether retakeable)
- **October:** PTA locked at October 1; no structural changes allowed without special approval
- **October–May:** Teachers enter SE (school exam) scores into the CijferRegister keyed to PTA items; system auto-calculates SE final grades
- **May–June:** Central exam (CE) held; examiner scores entered by exam secretary with dual-entry verification
- **June:** Slaag-zakregeling applied per candidate per education level; retake deadlines enforced
- **July:** Retakes held and scored
- **August:** Diploma batch generated by exam secretary; examinee committee reviews and approves all candidates; final diplomas issued
- **September:** DUO transmission with return confirmation tracking

**Constraints:**
- **Wettelijk:** Every rule in Eindexamenbesluit VO is mandatory. Slaag-zakregeling differs by education level (VMBO basis/kader/gemengd/theoretisch, HAVO, VWO); kernvak rules (max 1 fail for HAVO/VWO) are non-optional.
- **Audit:** Every score, every recalculation, every PTA change must be auditable — reason, user, timestamp, before/after snapshot.
- **Learner privacy:** All score data covered by AVG (GDPR); learners must be able to view their own scores via pipelinq leerlingportaal.
- **DUO integration:** DUO BRON/ROOD format is strict; missing or wrong data causes rejection; return messages must be logged.
- **Long-term archive:** Diplomas and complete exam dossiers must be unaltered for 50 years; integrity verifiable via hash chain.

## Goals / Non-Goals

**Goals:**
- Implement complete CE/SE exam administration per Eindexamenbesluit VO
- PTA versioning with lock-after-October-1 automatic enforcement
- Dual-entry verification for CE scores; no single-person data entry into final grades
- Slaag-zakregeling logic per education level (kernvak rule, CE average rule, compensation rule) with visible reasoning
- Herkansing request workflow with deadline enforcement (3 working days for CE retake offer after 1st round)
- Highest-score rule: if retake taken, higher of original or retake is used for final grade
- Diploma batch generation with examinee committee second-review before issuance
- DUO transmission via openconnector DUO adapter; return-confirmation tracking
- 50-year audit trail with cryptographic integrity for diploma appeals
- Learner portal integration (view own scores, retake requests) via pipelinq
- Examensecretaris dashboard showing exam readiness, missing scores, retake queue

**Non-Goals:**
- Conflict resolution between school PTA and national Cevo (central exam authority) — assume Cevo exam dates/topics are imported separately
- Custom education levels beyond VMBO/HAVO/VWO/MBO — spec assumes Dutch national curriculum
- Real-time DUO validation (DUO SOAP service is async batch)
- Automated appeals processing — appeals route to decidesk for decision-maker review
- Multi-language: spec covers Dutch regulatory context; UI translation deferred to i18n phase

## Decisions

**D1: PTA as versioned first-class entity, not embedded in CijferRegister**
The PTA determines what can be scored. Teachers must only be offered PTA items. If PTA changes mid-year, old and new versions must coexist for audit. Decision: store PTA as separate entity with `versie`, `vastgesteld_op`, `vastgesteld_door` fields. Auto-lock logic: if `vastgesteld_op` < October 1 of school year, reject any structural changes (weighting, new items) without special approval.

**D2: Dual-entry verification for CE, not SE**
CE (central exam) is national and high-stakes — entered once, externally determined. SE (school exam) is continuous assessment by teachers — trusting individual entry. Decision: `CeResultaat` requires two independent entries (or one entry + one verification step by different user) before marking definitief; `CijferRegister` for SE can have single-teacher entry but `status: concept` until teacher marks definitief.

**D3: Slaag-zakregeling logic in Service layer, parameterized by education level**
The kernvak rule, CE-average rule, and compensation rule differ by education level. Rather than embed conditionals in the database, Decision: build `SlaagBerekenService` with pluggable rules per `onderwijssoort` + `examenNiveau`. Each rule returns (voldaan: bool, details: {}) so the UI can show reasoning.

**D4: Herkansing as separate entity + workflow**
Learners request retakes via portal; examensecretaris approves/denies based on eligibility. Decision: `HerkansingsAanvraag` entity tracks request, approval, outcome. Deadline enforcement: if aanvraagdatum > 3 work days after cijfer publicatie, system rejects unless admin override. Status flow: ingediend → toegekend/afgewezen → uitgevoerd (once retake taken and scored).

**D5: Highest-score rule: store both original + retake, use highest**
Transparency + appeal-ability requires keeping both scores. Decision: `CijferRegister` has `herkansing_van_cijfer_id` fk to original; `SeEindcijfer` computation uses MAX(original, retake). Reasoning field explains which was chosen.

**D6: Diploma batch generation as async Job**
Batch generation of 50+ PDFs is I/O expensive. Decision: exam secretary triggers batch job via UI; job generates all diplomas to "ready-for-review" state; examinee committee reviews in parallel via a dedicated approval workflow before PDF finalization and DUO transmission.

**D7: DUO transmission via existing openconnector DUO adapter**
Rather than build custom DUO SOAP client, Decision: use openconnector DUO adapter (already available in Scholiq). Service layer assembles `Diploma` + `Eindcijfer` data, calls DUO adapter, receives return messages, logs confirmation per diploma with timestamp.

**D8: 50-year archive with cryptographic integrity, via docudesk**
Rather than build archival logic in Scholiq, Decision: after diploma issuance, export full exam dossier (PTA-version, all CijferRegister entries, CE results, Slaag-zakberekening snapshot, Diploma) as a serialized bundle; send to docudesk for long-term storage with hash chain. Later verification API (diplmaCheck) provided by docudesk.

## Reuse Analysis

- **OpenRegister ObjectService** for all CRUD (ExamenKandidaat, PTA, CijferRegister, etc.) — no custom mappers needed
- **ObjectService.lockObject()** for PTA auto-lock after October 1
- **AuditTrailService** for tracking all score entries, changes, user+timestamp snapshots
- **FileService** for diploma PDF storage and download
- **ImportService/ExportService** for DUO data import and export
- **NotificationService** for deadline reminders (herkansing deadlines, DUO transmission status)
- **Pinia stores** via `createObjectStore` for exam candidate list, retake queue, dashboard state
- **CnDataTable, CnDetailPage** for UI of score entry, PTA view, diploma batch review
- **CnFormDialog** for PTA creation/edit, herkansing request forms

No new backend services beyond domain-specific Slaag-zakberekenService and DuoTransmissionService needed.

## Open Questions

- Q: Should the PTA-lock-after-Oct-1 rule be timezone-aware (school location) or UTC? → Decision: Use school timezone from scholiq organization config
- Q: If CE exam is delayed (e.g., exam secretary is ill), can October 1 PTA lock be deferred? → Decision: Admin override with audit trail. Deferred lock logged and flagged to inspectorate contact.
